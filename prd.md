# 🚗 基于录音数据的“车企视角”产品洞察提取引擎 - 需求说明文档 (V1.0)
## 一、 项目背景与战略定位
### 1. 战略诉求：从“执行服务商”向“咨询赋能方”转型
为定期向主机厂（赛力斯）输出深度产品洞察，证明我司数字化运营能力。通过真实的录音数据反哺车企产品优化与销售管理，提升合作中的主动权和议价能力。
### 2. 解决现状痛点：告别泛化分析，保护研发资源
*   **缺乏数据支撑**：既往汇报依赖个案故事，缺乏真实录音数据佐证。
*   **分析结论泛化**：传统的“嫌贵”、“换车”等战败归因，对车企产品迭代毫无指导价值。
*   **试错成本高**：直接让研发跑全量数据，极易产出“模棱两可”的结论，造成研发资源浪费。
### 3. 核心原则：业务先验，工程后置
**产品先跑通小样本验证，研发后置工程化放大。** 确保每一份研发投入都有确定性回报，杜绝“大干快上”导致的烂尾工程。
---
## 二、 核心设计理念与逻辑链路
本项目的本质不是“录音摘要”，而是构建一套**基于 SOP 对齐的语义解析与商业情报系统**。核心逻辑为“三级跳”：
1.  **一级跳（结构化事实）**：ASR 转写 + 业务元数据挂载（门店、销售、车型、成交结果），构建大宽表底座。
2.  **二级跳（语义对齐与归因）**：将对话切片，强行映射到《销售话术大纲》节点上，提取带有“逐字原文证据”的具体决策因子。
3.  **三级跳（风格识别与洞察聚合）**：基于对话特征提取客户“决策风格/人格特质”，并与决策因子进行交叉差值分析，生成高优洞察。
---
## 三、 系统数据架构设计（星型模型）
为兼顾“多维快速统计”和“细节原文溯源”，摒弃单一超大宽表，采用**“主-从-语义-画像”**四层存储架构：
### 1. `audio_master` (录音主事实表)
*   **颗粒度**：单条录音。
*   **核心字段**：`record_id`、门店信息、销售信息、客户基础画像(性别/年龄/企业或个人)、车型(M7/M8/M9)、产品类型(延保/轻改)、**成交/战败结果**。
*   **用途**：BI 看板的宏观转化率统计。
### 2. `audio_events` (对话切片明细表) —— 【核心产出】
*   **颗粒度**：录音中的关键对话事件片段。
*   **核心字段**：
    *   `event_id` / `record_id`
    *   `node_id`：对应 SOP 话术节点（如 N02_焕新权益解释）。
    *   `evidence_text`：**逐字摘录的原文证据（防幻觉硬约束，必须能在原文中找到）。**
    *   `sentiment`：情绪。
    *   `is_turning_point`：是否为决策转折点。
    *   `decision_factor`：具体决策因子（如：对固态电池折旧费界定产生严重疑虑）。
    *   `suggestion`：给车企的具体优化建议。
*   **用途**：下钻分析，点击“为何战败”时直接拉出原话。
### 3. `semantic_pool` (向量语义池)
*   **颗粒度**：高价值的客户异议/质疑片段。
*   **核心字段**：`embedding` 向量、关联的 `record_id`、`node_id`。
*   **用途**：对海量“嫌贵”、“条款不清”进行语义聚类，发现潜在的共性子痛点（如区分“嫌裸车贵”还是“嫌权益包贵”）。
### 4. `customer_psychographic` (客户心理与风格画像表) —— 【进阶亮点】
*   **颗粒度**：单次接待的客户。
*   **核心字段**：
    *   `decision_style_tags`：决策风格标签数组（如：[风险厌恶型, 证据要求型, 家人意见主导型]）。
    *   `big5_scores`：大五人格(OCEAN)预估分数及依据原话。
    *   `talk_ratio_customer`：客户话权比（行为特征）。
*   **用途**：高阶交叉洞察（如：高神经质客群在特定条款下的流失率分析）。

## 四、 核心解析方法论（AI 认知引擎设计）
引擎分为三个递进的解析模块，研发需按此逻辑设计 Pipeline：
### 模块 A：SOP 锚点与决策因子提取（解决“为什么败”）
*   **逻辑**：将业务定义的《销售话术大纲》作为强制坐标轴，让大模型做阅读理解。
*   **硬性约束**：
    1.  **拒绝泛化**：禁止输出“不需要”、“嫌贵”，必须指向具体条款。
    2.  **证据闭环**：必须输出 `evidence_text`，且研发需引入相似度校验逻辑（低于85%相似度视为幻觉，打入异常库）。
### 模块 B：客户决策风格识别（解决“什么类型的人败”）
*   **逻辑**：不依赖心理学量表，直接从对话行为中打标签。
*   **识别维度**：价格敏感型、风险厌恶型（频繁追问最坏情况）、技术极客型、证据要求型（索要合同截图/白纸黑字）等。
*   **特征来源**：部分通过大模型提取（如索要证据），部分通过 ASR 统计特征提取（如话权比、提问密度）。
### 模块 C：深层人格特质推断（探索性模块，用于群体分析）
*   **逻辑**：采用“角色扮演 + 问卷式 Prompt”，让 LLM 以对话为依据，为客户填写简化版大五人格（OCEAN）问卷。
*   **边界约束**：明确声明此分数仅用于**群体特征差异分析**（如：高开放性人群对固态电池接受度更高），绝对禁止用于对单个客户的歧视性标签或直接外发。

## 五、 洞察生成与业务应用场景（Insight Engine）
基于上述四张表，在 BI 层进行差值与交叉分析，直接生成月度汇报素材：
1.  **产品条款痛点洞察（给赛力斯产品部）**：
    *   *分析*：战败订单中，`audio_events` 表里 `node_id=N03(固态电池条款)` 且 `is_turning_point=True` 的高频 `decision_factor` 是什么？
    *   *产出*：“经分析，‘固态电池工时费承担边界模糊’是打断购买意愿的最高频转折点，建议修改权益白皮书。”
2.  **销售能力诊断（给门店督导）**：
    *   *分析*：某销售成交率低，关联 `audio_events` 发现其记录中缺失 `N02(焕新权益)` 节点。
    *   *产出*：“该销售在推介中跳过核心权益讲解，直接导致后续价格战败，需专项复训。”
3.  **客群风格与话术适配洞察（高阶产出）**：
    *   *分析*：交叉 `customer_psychographic` (风险厌恶型) 与 `audio_events` (战败节点)。
    *   *产出*：“‘风险厌恶型’客户在听到‘免责条款’时流失率比均值高40%，建议针对此类客群前置强化‘兜底保障’话术。”

## 七、 附录：业务侧 MVP 手动验证操作手册
*(本手册为 Phase 1 的标准执行动作，业务分析师需严格按此步骤执行，以确定最终的 Prompt 与数据标准)*
### 目标与产出
用 5–10 条 ASR 文本在 Excel 内跑通链路，核心产出 `audio_events` 明细事件表，每条事件必须有原文证据。
### 操作步骤
#### 第一步：业务手搓“SOP 标签体系”
在 Excel 的 `Sheet_Config` 中建表（以延保为例，禁止超过 8 个节点）：
| node_id | node_name (话术节点) | node_description (节点说明) |
| :--- | :--- | :--- |
| N01 | 延保价值引入 | 销售初步抛出延保概念，强调三电终身质保的兜底作用。 |
| N02 | 焕新权益解释 | 重点解释“多少年/多少公里后可触发焕新”，以及触发条件的界定。 |
| N03 | 固态电池条款 | 解释未来固态电池升级时的费用承担规则（如工时费、折旧费）。 |
| N04 | 价格异议处理 | 客户觉得贵，销售通过算账、分摊等方式化解。 |
| N05 | 逼单成交 | 销售利用限时政策、店总特权等施加紧迫感。 |
#### 第二步：构建模拟数据库
*   **Sheet 1 `audio_master`**：填入 `record_id` | `store_name` | `sales_name` | `car_model` | `product_type` | `deal_result`。
*   **Sheet 2 `audio_events`**：建好表头：`record_id` | `event_id` | `node_id` | `customer_quote` | `evidence_text` | `sentiment` | `is_turning_point` | `decision_factor` | `suggestion`。
#### 第三步：执行大模型对齐提取
**切记：不要把长录音一次性丢给模型，按自然段切分。** 使用以下强制约束 Prompt：
```text
# Role
你是资深车企产品体验分析师。你的任务是将提供的【录音转写文本】，精准映射到我提供的【SOP节点】上，并提取客户的“决策因子”。
# SOP 节点字典
{N01: 延保价值引入, N02: 焕新权益解释, N03: 固态电池条款, N04: 价格异议处理, N05: 逼单成交}
# 极限约束规则 (违反则任务失败)
1. 拒绝泛化：禁止输出“嫌贵”“不需要”“再看看”等空泛结论。必须写清楚：客户针对哪个具体条款/哪句话产生了什么顾虑或推动。
2. 证据闭环：`evidence_text` 必须从原文中逐字复制，标点符号不准改；找不到就填“无”。
3. 不要编造：如果原文中没有明确证据支撑 `decision_factor`，则该条事件不要输出。
4. 聚焦转折：只有导致客户态度发生根本转变（如从有意向变冷淡、从犹豫变下定决心）的对话，`is_turning_point` 才能为 true。
5. 强制 JSON：只输出 JSON 数组，不要任何解释。
# 输出 Schema
[
  {
    "event_id": "E001",
    "node_id": "N02",
    "customer_quote": "客户原话逐字摘录",
    "evidence_text": "逐字证据，找不到填 无",
    "sentiment": "Positive / Neutral / Negative",
    "is_turning_point": true,
    "decision_factor": "必须具体且可复核，例如：对焕新触发条件中的‘电池衰减必须大于30%’感到无法接受",
    "suggestion": "给车企的建议，必须可执行，避免空话"
  }
]
# 录音转写文本：
[在此处粘贴你的切分后ASR文本]
```
将返回的 JSON 解析填入 `audio_events` 表，重复跑完 5-10 条。
#### 第四步：Go/No-Go 验收标准（硬卡点）
跑完后，用以下标准决定是否向研发提交工程化需求：
1. **证据可复核（一票否决）**：抽检 `evidence_text`，能在原文一字不差找到。若 20% 不通过，No-Go。
2. **因子足够具体（一票否决）**：`decision_factor` 若仍是“价格敏感/观望情绪”占比>30%，说明 Prompt 失败，No-Go。
3. **建议可执行**：`suggestion` 至少有 3 条能被车企产品部直接接住（能改条款/改话术）。
**拿着这份 Excel 跑出来的至少 3 条硬核洞察（带原文证据）去评审，研发即可直接据此落库结构与接口链路。**

## 第五步：Go/No-Go（验证卡点与验收标准）
跑完 5–10 条后，用以下标准做结论：

## 实操小贴士
- 不追求 100%：10 条里 6–7 条能产出高质量 `decision_factor` 即可证明业务逻辑成立。
