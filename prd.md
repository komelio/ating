# 🛠️ 洞察引擎 MVP（手动验证 → 可工程化）操作手册

## 目标与产出
- **目标**：用 5–10 条 ASR 文本在 Excel 内跑通一套“可复现”的洞察生产链路，验证业务价值与工程可行性。
- **核心产出物**：`audio_events` 明细事件表（每条事件必须有原文证据）、以及基于它的差值洞察（成交 vs 战败）。

## 适用范围与输入约束
1. **样本**：5–10 条完整 ASR 转写文本。
   - 覆盖：至少 3 条战败 + 2 条成交。
   - 必须包含：销售推销“延保/轻改”等具体产品的过程。
2. **ASR 文本要求**：尽量保留说话人分隔（如“销售：/客户：”）；若没有说话人，也可先执行，但需在事件表里补充 `speaker` 字段以降低歧义。

---

## 第一步：业务手搓 “SOP 标签体系”（模拟 Rule Layer）
不要搞太复杂，针对你要验证的“产品（如延保）”，手动写出 5–8 个具体节点。

**操作：在 Excel 的 `Sheet_Config` 中建表如下：**

| node_id | node_name (话术节点) | node_description (节点说明，用于喂给大模型) |
| :--- | :--- | :--- |
| N01 | 延保价值引入 | 销售初步抛出延保概念，强调三电终身质保的兜底作用。 |
| N02 | 焕新权益解释 | 重点解释“多少年/多少公里后可触发焕新”，以及触发条件的界定。 |
| N03 | 固态电池条款 | 解释未来固态电池升级时的费用承担规则（如工时费、折旧费）。 |
| N04 | 价格异议处理 | 客户觉得贵，销售通过算账、分摊等方式化解。 |
| N05 | 逼单成交 | 销售利用限时政策、店总特权等施加紧迫感。 |

---

## 第二步：构建“模拟数据库”（模拟 Storage Layer）
打开 Excel，新建两个 Sheet，这就是未来研发要建的“主表”和“明细事件表”。

### Sheet 1：`audio_master`（录音主宽表）
手工填入基础信息：

*列：`record_id` | `store_name` | `sales_name` | `car_model` | `product_type` | `deal_result`*

### Sheet 2：`audio_events`（对话事件明细表 - 核心产出）
先建好表头，等待大模型产出来填入。

建议表结构升级为（新增字段用于工程化与校验）：

*列：*
- `record_id`
- `event_id`（同一录音内唯一即可，如 `E001`）
- `node_id`
- `speaker`（Customer / Sales / Other / Unknown）
- `customer_quote`（保持原字段名以兼容现有流程；但当 speaker≠Customer 时允许为“无”）
- `evidence_text`（逐字摘录的原文证据；可以与 customer_quote 相同，但建议单独保留，便于后续扩展多句证据）
- `evidence_start_hint`（可选：证据在原文中大致位置，如“第3段”或“约 02:10”，手工阶段可为空）
- `sentiment`（Positive / Neutral / Negative）
- `is_turning_point`（是否决策转折）
- `decision_factor`（具体因子）
- `suggestion`（产品建议）

---

## 第三步：执行“大模型对齐提取”（模拟 AI Cognition Layer）
这是最关键的一步。不要把很长的录音一次性丢给模型。

### 推荐操作方式（手动阶段的最小工程化）
1. 每次取 **1 条录音文本**。
2. 如果单条录音文本过长，按自然段/轮次切为 2–4 段分别跑（每段都带上同一套 SOP 节点字典），最后把结果合并到同一个 `record_id`。

### 强制约束 Prompt（优化版）
```text
# Role
你是资深车企产品体验分析师。你的任务是将提供的【录音转写文本】，精准映射到我提供的【SOP节点】上，并提取客户的“决策因子”。

# SOP 节点字典
{N01: 延保价值引入, N02: 焕新权益解释, N03: 固态电池条款, N04: 价格异议处理, N05: 逼单成交}

# 极限约束规则 (违反则任务失败)
1. 拒绝泛化：禁止输出“嫌贵”“不需要”“再看看”等空泛结论。必须写清楚：客户针对哪个具体条款/哪句话产生了什么顾虑或推动。
2. 证据闭环：`evidence_text` 必须从原文中逐字复制，标点符号不准改；找不到就填“无”。
3. 不要编造：如果原文中没有明确证据支撑 `decision_factor`，则该条事件不要输出。
4. 聚焦转折：只有导致客户态度发生根本转变（如从有意向变冷淡、从犹豫变下定决心）的对话，`is_turning_point` 才能为 true。
5. 强制 JSON：只输出 JSON 数组，不要任何解释。

# 输出 Schema
[
  {
    "event_id": "E001",
    "node_id": "N02",
    "speaker": "Customer",
    "customer_quote": "客户原话逐字摘录；如果 speaker 不是 Customer 则填 无",
    "evidence_text": "逐字证据，可为一到两句原文；找不到填 无",
    "sentiment": "Positive / Neutral / Negative",
    "is_turning_point": true,
    "decision_factor": "必须具体且可复核，例如：对焕新触发条件中的‘电池衰减必须大于30%’感到无法接受",
    "suggestion": "给车企的建议，必须可执行，避免空话"
  }
]

# 录音转写文本：
[在此处粘贴你的第1条ASR文本]
```

3. 把大模型返回的 JSON 数组手动解析（或用代码解释器转表格），贴入 `audio_events` 中，并关联对应的 `record_id`。
4. 重复此动作，跑完 5–10 条样本。

---

## 第四步：在 Excel 中“生产洞察”（模拟 Insight Engine）
现在你的 Excel 里有了模拟数据，用数据透视表做“差值分析”。

### 实操演练 1：挖掘“产品条款”痛点（给车企看的干货）
1. 选中 `audio_events` 表，插入数据透视表。
2. **行**：拖入 `decision_factor`。
3. **列**：拖入 `deal_result`（成交/战败）。
4. **值**：拖入 `is_turning_point`（计数）。
5. **洞察产出**：例如你看到在“战败”列里，`N03: 固态电池条款` 的 `decision_factor` 高发且 `is_turning_point=true`。
   - 报告话术示例：“经抽取本月战败录音分析发现，‘固态电池工时费承担边界’是打断客户购买意愿的最高频转折点（占比60%），建议产品部审视该条款的透明度与前置解释策略。”

### 实操演练 2：挖掘“销售能力”短板（给门店看的干货）
1. 将 `audio_master` 和 `audio_events` 用 VLOOKUP 连接起来。
2. 筛选某一个销售的记录。
3. **洞察产出**：例如你发现某销售的记录里，几乎没有 `N02: 焕新权益解释` 的节点标签。
   - 报告话术示例：“该销售在延保推介中高频跳过‘焕新权益’讲解，直接进入价格谈判，导致在价格环节遭遇重挫。需针对 N02 节点专项复训。”

---

## 第五步：Go/No-Go（验证卡点与验收标准）
跑完 5–10 条后，用以下标准做结论：

1. **证据可复核（硬门槛）**：`evidence_text` 能在原始 ASR 文本中一字不差找到；若抽检 20% 事件不通过，则 No-Go。
2. **因子足够具体（硬门槛）**：`decision_factor` 能指向明确条款/话术/条件边界；若仍是“价格敏感/观望情绪”等泛化词占比>30%，则 No-Go。
3. **建议可执行（软门槛）**：`suggestion` 至少有 3 条能被产品/运营明确接住（能改条款/改宣导/改话术/改培训），否则需要改 Prompt 角色与约束。

---

## 实操小贴士
- 不追求 100%：10 条里 6–7 条能产出高质量 `decision_factor` 即可证明业务逻辑成立。
- 保留“脏数据”：刻意加入 1 条噪音大/跑题严重的录音，验证模型是否会输出空数组而不是胡编乱造。
- 事件粒度别贪：宁可少抽取也要证据闭环，避免后续工程化时“不可复核”的数据变成技术债。

**拿着这份 Excel 跑出来的至少 3 条硬核洞察（带原文证据）去评审，研发就能直接据此落表结构、接口与处理链路。**
