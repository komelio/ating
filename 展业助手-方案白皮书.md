## 一、需求梳理

### 1.1 项目背景与目标

「展业助手 喵加 SAAS」是广联数科面向汽车后市场门店的一体化展业与运营平台。当前系统为单体 Dubbo RPC 架构（约 250 个 RemoteImpl、450 个 Service、449 个 Mapper、128 个 ElasticJob），存在模块边界模糊、部署粒度过粗、缺少自有前端等痛点，亟需技术重构与功能升级。

**目标客户**：4S 店（延保/精品销售）、洗美店（服务展业）、后服务商（维修/保养/改装）

**核心目标**：
1. 五大上架体系化管理，标准化门店准入与运营全流程
2. 银盛/通联支付深度集成，自动收单→回调→自动分账，消除人工对账
3. 外部生态打通：北森 HR 同步、大模型 AI、K3 财务编码同步
4. 多端覆盖：PAD 展业助手（驻店销售）+ Web 管理后台（运营/财务）构成 MVP 核心

### 1.2 四大优化升级

| 模块 | 优化前 | 优化后 |
|------|--------|--------|
| 客户触达 | 依赖接待分配系统 | 无接待分配，谁录单归谁 |
| 支付收单 | 手动录流水，人工匹配 | 银盛扫码支付，自动回调 |
| 分账清分 | 离线 Excel + 人工打款 | 分账模板自动分账，资金直达 |
| 产品编码 | 系统自定义编码 | SKU 绑定 K3 编码，财务同步 |
| 前端 | 依赖 SCRM Proxy | 自建 Vue3 运营端 + 移动端 |

### 1.3 五大上架体系

系统的核心运营引擎，定义门店/人员/场景/产品/商品进入系统的标准与流程：

```
┌──────────────┐     ┌──────────────┐
│ 1.场景上架    │     │ 2.产品上架    │
│ 拓店/展业/   │     │ SPU/SKU库    │
│ 闭店标准管理  │     │ + K3编码绑定  │
└──────┬───────┘     └──────┬───────┘
       └────────┬───────────┘
                ▼
        ┌──────────────┐
        │ 3.门店上架    │  签约准入 + 银盛MID + 商务政策
        └──────┬───────┘
         ┌─────┴─────┐
         ▼           ▼
  ┌──────────┐  ┌──────────┐
  │4.商品上架  │  │5.人员上架  │
  │ 门店定价  │  │ 人才库+绩效 │
```

## 二、关键业务流程

### 2.1 整体架构分层

```
表现层(PAD展业助手/Web后台/BI看板) → 业务中台(12大服务域) → 上架及账户中心(底座) → 外部生态
```

### 2.2 核心交易闭环（最大业务价值）

```
驻店销售录单 → 选择门店商品 → 生成订单 → 银盛支付台卡 → 车主扫码支付
    → 银盛回调成功 → 更新订单PAID → 自动分账(门店70%/销售15%/服务商10%/平台5%)
    → 触发施工通知 → 技师扫码完成 → 订单关闭
```

**银盛回调处理流程**：
1. 银盛 POST 回调 → 验签（证书+签名）
2. 匹配订单号+金额，查 ys_payment_orders
3. 数据库事务：orders=PAID, payment_orders=PAID, 写入回调报文全量
4. 异步触发分账：查分账模板 → 调用银盛分账API → 资金直达各方银盛电子账户

### 2.3 门店生命周期

```
签约中(SIGNING) ──MID审核通过──▶ 展业中(OPERATING) ──触发闭店条件──▶ 已闭店(CLOSED)
```

### 2.4 清分对账流程

```
运营/财务 → 查看对账看板(按门店/时间筛选) → 生成对账报告(银盛回调数据分组汇总)
    → 导出 Excel(门店/服务商维度) → 导出 K3 格式对账单(SKU级编码对齐)
```

### 2.5 流程间数据依赖

```
场景上架 → 门店上架 → 人员上架 ┐
         ↘ 产品上架 → 商品上架 ┘→ 交易收单(银盛支付+分账) → 清分对账(导出K3)
```

## 三、技术方案和选型

### 3.1 架构策略

**一期：模块化单体**，Maven Module 划分订单/商品/结算/报表域，强制模块间 API 契约。
**二期按需**拆微服务。理由：450+ Service 调用关系复杂，直接拆分风险极高。

### 3.2 技术栈全景

```
┌─────────────────────────────────────────────────────────────┐
│                      前端层                                  │
│  运营Web端：Vue3+TS+ElementPlus+Vite+Pinia+ECharts          │
│  移动端H5：Vue3+TS+Vant4+Vite+Pinia (Pad/小程序/App通用)     │
│  工程化：pnpm+Turborepo+OpenAPI Generator→TS类型             │
├─────────────────────────────────────────────────────────────┤
│                      网关层：Nginx + JWT 鉴权                 │
├─────────────────────────────────────────────────────────────┤
│                    应用层：Spring Boot 3.x + Java 17          │
│  [order] [goods] [settle] [report] [system] ...             │
│  模块API层 + 共享设施：MyBatis-Plus + Actuator + OpenAPI     │
├─────────────────────────────────────────────────────────────┤
│                    基础设施层                                 │
│  MySQL8(主从) | Redis7+Caffeine | XXL-JOB | RabbitMQ       │
│  阿里云OSS/MinIO | Dubbo3(内部RPC) | Prometheus+Grafana     │
├─────────────────────────────────────────────────────────────┤
│                    DevOps：Docker Compose → K8s              │
└─────────────────────────────────────────────────────────────┘
```

### 3.3 核心选型对比（与旧系统）

| 维度 | 旧系统 | 新系统 | 理由 |
|------|--------|--------|------|
| Java | Java 8 | Java 17/21 | 虚拟线程简化异步 |
| 模块划分 | 无模块，Service 随意调用 | Maven Module + API 契约 | 编译期约束边界 |
| ORM | MyBatis + 449 XML | MyBatis-Plus | 消除 60% XML |
| JSON | fastjson | Jackson | fastjson 已停维 |
| 调度 | ElasticJob(128 Job) | XXL-JOB | 可视化管控+失败告警 |
| 鉴权 | 依赖外部系统 | JWT + RBAC 内置 | 独立权限体系 |
| 前端 | 无(依赖 SCRM Proxy) | Vue3 自建两套 | 全链路自控 |
| API 协议 | Dubbo(对内) | RESTful(网关) + Dubbo(对内) | 前端直调 |
| 消息队列 | ActiveMQ | RabbitMQ | 更可靠+延迟消息 |
| 部署 | WAR 包 | Docker 镜像 + Compose | 容器化 |
| 可观测 | Druid + Dubbo Metrics | Micrometer + Prometheus | 统一指标 |

### 3.4 RBAC 权限模型

```
账号(t_account) → 门店角色关联(t_account_store_role)
    ├── store_id   → 数据权限：能看哪些门店
    ├── role_id    → 功能权限：能操作哪些菜单/按钮
    └── data_scope → 数据范围：本店/下级/辖区
```

### 3.5 权限角色矩阵

| 功能域 | 超级管理员 | 平台运营 | 店长 | 销售顾问 | 财务 |
|--------|-----------|----------|------|----------|------|
| 订单管理 | ● CRUD | ● 查看+处理 | ● 本店 | ● 个人 | ● 退款审批 |
| 商品管理 | ● CRUD | ● CRUD | ○ 上架 | — | — |
| 结算对账 | ● | ○ 查看 | ○ 查看 | — | ● CRUD |
| 门店管理 | ● CRUD | ● 查看+编辑 | — | — | — |
| 系统管理 | ● | — | — | — | — |

●全部 ○部分 —无权限

### 4.3 纯AI开发模式说明

本项目采用 **CodeBuddy AI Agent 集群** 进行纯 AI 开发，核心模式：

- **AI 并行生成**：前端/后端/数据库任务由多个 AI Agent 同时并行推进，互不阻塞
- **人工 Review**：1 名技术架构师 + 1 名业务/产品对接人负责需求澄清、代码审核、集成测试
- **自动化验证**：Vitest/JUnit 单元测试 + Playwright E2E 测试由 AI 自动生成并执行
- **CI/CD 自动化**：Git 提交 → 自动构建 → 镜像推送 → Docker Compose 部署
